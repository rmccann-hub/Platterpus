> Produced by an exhaustive multi-agent review (13 code/doc auditors + 3 web-research

## A. Executive summary

**Overall health: strong core, with a systemic Qt-threading flaw and default/edge gaps in the newest features.** The codebase is genuinely well-built — 93% branch coverage, ruff-clean, real adapter seams, verified-clean layer discipline, a fail-safe CTDB seam that holds, and a disciplined docs culture. The problems cluster in three places: (1) a **recurring Qt-threading dispatch bug** — work meant to run on a worker actually runs on the GUI thread, and several close/cancel paths join threads on the GUI thread — which the AST fitness guard *structurally cannot catch*; (2) the **0.4.9 headline features** (dynamic secure re-rip, per-track auto-fix) ship with a default that makes them inert and an edge case that misbehaves on the exact disc class the maintainer tests with; and (3) **pervasive whipper→cyanrip doc drift** (whipper was removed 2026-06-30 but the architecture guide, PLANNING §5, and several rules still describe it as current).

**Top 5 risks**
1. **MusicBrainz lookups run on the GUI thread** (confirmed by hand). The MB worker is moved to a thread that only idles; the queries are invoked as direct calls, so they run on the GUI thread — and the MB adapter sets no network timeout. A slow/stalled MusicBrainz freezes the window on the *most common* action (inserting an identified disc).
2. **The dynamic secure re-rip headline is inert by default, and harmful when enabled on a not-in-database disc.** `secure_rerip_matches` defaults to 0 (and all presets set 0), so the feature never activates on a fresh install — yet the CHANGELOG says it does. When a user does set `-Z`, a disc not in AccurateRip (e.g. a CD-R — the maintainer's own first test disc) makes it re-rip the *entire* disc a second time and swap every FLAC.
3. **Silent wrong-metadata / cross-album contamination.** MusicBrainz results have no staleness guard, so a late lookup from the previous disc can tag the new disc as the wrong album; and async post-rip verifiers (CTDB/FLAC/transcode) have no rip-identity guard, so album A's slow verify can write its verdicts into album B's report and UI. This attacks the maintainer's north star directly — a trustworthy library entry.
4. **A single invalid-UTF-8 byte in a rip log crashes the finish handler and aborts the whole post-rip chain** — no JSON report, no tagging, no cover art, no transcode, no eject, and the rip state is never cleared (so close-on-shutdown then treats a finished rip as live).
5. **The never-block-the-GUI-thread rule keeps being re-violated in new code because its guard is blind.** The AST guard only scans `ui/` and can't follow a slot into a blocking helper — exactly how the check-dependencies freeze, the drive-Refresh freeze, and the wizard-close freeze all shipped past it.

**Single highest-value change:** fix the MusicBrainz off-thread dispatch + add the missing adapter timeout (risk 1), and in the same effort **extend the GUI-thread guard to catch the indirection class** (risk 5). That removes the most-hit freeze *and* closes the structural hole that let this bug class recur three-plus times.
